# Datapaths
## Core Idea
A datapath is the hardware that moves and process data. It is made from things like:
- Registers
- Bus 
- Multiplexers/Tri-state buffers
- ALU/Adder
- Control Signals
Example:
You want to add four 8-bit numbers, but you only have one adder, and each adder can only take two inputs. So instead of building four adders, you build a system that moves values through the same adder over multiple clock cycles. That system is the datapath.

## Bus
A **bus** is a shared group of wires used to transport data. In the slides, the bus connects several registers to shared hardware like the adder or ALU. The important point is:

> A bus is not magic memory. It is just wires.

Only one component should drive the bus at a time. If two registers both try to output onto the same wire, one might drive `0` while another drives `1`. In simulation this usually appears as `X`, meaning unknown. In real hardware, this can cause electrical fighting because one circuit is trying to pull the wire high while another pulls it low.

## Tri-state buffers
A **tri-state buffer** solves the “multiple outputs connected to one wire” problem.
It has three possible output behaviours:

|Enable|Output|
|---|---|
|`1`|Drives `0`|
|`1`|Drives `1`|
|`0`|High impedance, written as `Z`|

`Z` means the output is effectively disconnected, like an open circuit. Your slides also point out that `X` and `Z` are simulation values, not physical voltages you can directly measure in the lab. 

In Verilog, this idea looks like:
```verilog 
assign bus = enable ? data : 8'bz;
```
But for FPGA-style processor design, you often avoid internal tri-state buses and use multiplexers instead.

## Multiplexers as bus selectors
A **multiplexer** chooses one of many possible inputs.
Instead of having multiple registers directly drive a bus, you can use a MUX:
```verilog
assign bus =    sel == 2'b00 ? R0 :    sel == 2'b01 ? R1 :    sel == 2'b10 ? R2 :                   R3;
```
This means only one register’s value is selected at a time. Functionally, it does the same job as controlled bus access, but it is usually safer and clearer in Verilog.

## ALU
The later datapath slides add an ALU. The ALU can perform operations such as:
```
A + BA - Bmove data between registers
```
The adder/subtractor block works by using the fact that:
```
A - B = A + (-B)
```
To compute `-B` in two’s complement, invert all bits of `B` and add 1. The slides describe the `AddSub` signal as both the mux selector and the carry-in to the adder. 

So the ALU is not independent from the control unit. It needs control signals like:
```
AddSubAinGinGoutR0outR1outR2inDone
```


# Finite State Machine (FSM)
## Core idea
A **finite state machine**, or FSM, is a sequential circuit used when the system needs to remember what has happened before.
Your FSM slides define state as the information you need about the past to decide what to do next. For example, in a lift/elevator system, it matters what floor you are on, what direction you are moving, and which calls are pending. That information is the state. 
A general FSM has:

- Inputs
- Current State
- Next State Logic
- State Registers
- Output Logic
- Outputs

The state is stored in flip-flops or registers. On each clock edge, the FSM updates from the current state to the next state.
## FSM structure
The simplified structure in words:

```
current state + inputs → next state
current state, and maybe inputs → outputs
```

## FSM design process
Your FSM slides give a very practical design process:

|Step|What you do|
|---|---|
|1|Understand the problem|
|2|Decide what components are needed|
|3|Draw a state diagram|
|4|Decide how many states are needed|
|5|Assign binary codes to states|
|6|Create a state transition table|
|7|Translate into gates or Verilog|
|8|Create output logic|
|9|Put everything together|

The state diagram is a visual description of how the FSM moves from one state to another. The state transition table then converts that into a more hardware-friendly truth-table format. 

## Example: 3-bit counter
The first FSM example is a 3-bit counter.
It has states:

```
000 → 001 → 010 → 011 → 100 → 101 → 110 → 111 → 000
```

The output is just the current count, so the output logic is basically just a wire from the state register to the output. 

In Verilog terms, the FSM can be split into:

```verilog
// next-state logic
always @(*) begin
    case (state)
        3'b000: next_state = 3'b001;
        3'b001: next_state = 3'b010;
        3'b010: next_state = 3'b011;
        3'b011: next_state = 3'b100;
        3'b100: next_state = 3'b101;
        3'b101: next_state = 3'b110;
        3'b110: next_state = 3'b111;
        default: next_state = 3'b000;
    endcase
end

// state register
always @(posedge clk or posedge rst) begin
    if (rst)
        state <= 3'b000;
    else
        state <= next_state;
end

assign count = state;
```

This matches the slide structure: compute next state, store it in state registers, then use the state as output. 

## Example: mod-5 / count-to-5 FSM
The second example modifies the counter so it only counts up to a certain state, then returns to zero. The output becomes `1` when a certain state is reached. 

This teaches two important ideas:

1. Not every binary state needs to be used.
2. Unused states should be handled safely.

For example, if you only use states `000` to `101`, then states `110` and `111` are unused. In Verilog, you usually handle these with a `default` case so the FSM does not get stuck in an illegal state.

## Example: sequence detector
The sequence detector example looks for five `1`s in a row. This is a classic FSM because the circuit must remember how many consecutive `1`s it has seen so far.

The states mean something like:

|State|Meaning|
|---|---|
|A|Seen zero consecutive `1`s|
|B|Seen one consecutive `1`|
|C|Seen two consecutive `1`s|
|D|Seen three consecutive `1`s|
|E|Seen four consecutive `1`s|
|F|Seen five consecutive `1`s|

If the input `w` is `1`, the FSM moves forward. If `w` is `0`, it resets back toward the initial state. The slides then convert this state diagram into a state transition table and Verilog `case` statement. 

This example is important because it shows why FSMs exist: combinational logic alone cannot remember “how many 1s in a row have happened.”

## Moore vs Mealy machines
The textbook backup material helps clarify a key FSM distinction.

### Moore FSM
In a **Moore machine**, outputs depend only on the current state.

```
output = f(state)
```

Example:

```
assign detected = (state == FOUND);
```

The textbook notes that for a Moore FSM, output logic can be written as a combinational function of the present state. 

### Mealy FSM
In a **Mealy machine**, outputs depend on the current state and inputs.

```
output = f(state, input)
```

Example:

```
assign detected = (state == FOUR_ONES && w == 1'b1);
```

The textbook explains that in a Mealy machine, the output can change as soon as the input changes because the output is part of the combinational logic. 

# How Datapaths and FSMs connect
This is the most important connection for your processor work.
A datapath by itself is passive. It has registers, an ALU, and buses, but it does not know what to do.
An FSM by itself does not process useful data. It just moves between states and creates control outputs.
Together:

```
FSM = controllerDatapath = worker
```

For example, suppose the processor instruction is:

```
R3 ← R1 + R2
```

The datapath contains:

```
R1, R2, R3, bus, ALU
```

The FSM generates the sequence:

```
State 0: put R1 onto bus, load AState 1: put R2 onto bus, load BState 2: tell ALU to addState 3: write ALU result into R3State 4: done
```

So the FSM does not “add” values itself. It controls the datapath so the datapath performs the addition.
